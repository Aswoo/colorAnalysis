# Color Analysis 알고리즘 변천사

HSV 기반 단순 H 범위 비교에서 시작해 Lab 공간 K-means + LCH Hue 각도 기반 유사도 계산에 이르기까지, 각 단계에서 어떤 문제를 발견했고 왜 그 방향으로 개선했는지 기록합니다.

---

## 1단계: HSV H 범위

### 접근 방법

BGR 이미지를 HSV로 변환한 뒤, K-means 클러스터 중심의 H 값이 색상별로 미리 정의한 범위 안에 들어오는지 확인합니다.

```
K-means (BGR 공간)
    └─ 각 클러스터 → BGR → HSV
            ├─ S < 50 || V < 50 → 무채색으로 제외
            ├─ H 범위 매칭
            └─ 유사도: 100 - (|H - 범위중심| / 범위절반 × 30)
```

matched 조건:
1. 유사도 ≥ 70인 클러스터 면적 합 ≥ 이미지의 20%
2. 타겟 색이 유채색(S≥50, V≥50) 픽셀의 33% 초과

### 발견한 문제

**① 어두운 픽셀이 무채색 필터에 의해 제거됨**

`V < 50` 필터가 그림자 영역 전체를 날립니다. 노란 나뭇잎의 그늘진 부분이 갈색으로 인식되면서 황색 성분이 실제보다 적게 집계됩니다.

**② 저채도 픽셀의 PINK 오탐**

`yellow_tree` 이미지의 회색 도로·하늘 클러스터가 PINK로 판정됐습니다. S가 낮은 무채색 픽셀의 H 값은 잡음이 심한데, 이 값이 우연히 PINK 범위에 들어왔습니다. Lab Chroma 개념이 없어 걸러낼 수단이 없었습니다.

**③ 색상 범위 경계의 임의성**

PINK/RED, BLUE/PURPLE 등 인접 색 간 경계를 어디서 자를지 명확한 기준이 없었습니다. 특정 이미지 실패 시 해당 색의 범위를 늘리는 방식으로 대응했고, 이는 다른 색의 오탐 가능성을 높였습니다.

## 2단계: Lab ΔE (CIE76 거리)

### 접근 동기

HSV H만으로는 인간이 실제로 느끼는 색 차이를 반영하기 어렵다는 판단 하에 Lab 색공간으로 전환했습니다. 각 색상에 기준점(L, a, b)을 정의하고 K-means 클러스터 중심과의 CIE76 ΔE를 계산했습니다.

```
ΔE = sqrt((L1-L2)² + (a1-a2)² + (b1-b2)²)
유사도 = 100 - (ΔE / maxDeltaE) × 40
```

무채색 필터도 `S < 50` 대신 Lab Chroma(`sqrt(a² + b²) < 15`)로 교체했습니다. 덕분에 gray 픽셀의 PINK 오탐이 해소되었습니다.

### 발견한 문제

**① L(명도) 차이가 ΔE를 지배함**

`yellow_tree` 이미지의 어두운 황갈색 클러스터 분석:

```
클러스터 #865519  →  Lab L=41, a=10, b=35
YELLOW 기준점    →  Lab L=80, a=5,  b=45
ΔE = sqrt((41-80)² + (10-5)² + (35-45)²) ≈ 40.7
maxDeltaE = 40.0  →  경계 초과 → 탈락
```

사람 눈에는 "어두운 노랑(황갈색)"인 이 클러스터가 L 차이(39)로 인해 임계값을 초과했습니다. 이 문제는 maxDeltaE를 늘려 해결할 수 없습니다. 임계값을 크게 잡으면 YELLOW가 인접 GREEN, ORANGE까지 매칭합니다. **L이 색상 판별에 개입하면 안 된다**는 결론에 이르렀습니다.

## 3단계: LCH Hue 각도

### 핵심 아이디어

a, b 채널만으로 Hue 각도를 계산하고 L은 완전히 무시합니다.

```
hue = atan2(b, a)  →  0°~360° 정규화
```

```
밝은 노랑   a=5,  b=45  →  hue ≈ 83.7°
어두운 황갈 a=10, b=35  →  hue ≈ 74.1°
YELLOW 기준: hue=85°, maxHueDist=30°
→ 두 클러스터 모두 범위 내 → 매칭 성공
```

색상 정의가 `(refHue, maxHueDist)` 두 값으로 단순해졌습니다. Hue는 원형(0°=360°)이므로 거리는 최단 경로로 계산합니다.

```java
double diff = Math.abs(hue - refHue);
return Math.min(diff, 360.0 - diff);
```

밝기 가중치는 HSV V 대신 Lab L을 사용합니다. 완전히 어두운 클러스터는 weight를 낮추되, 색상 자체는 계속 인식합니다.

| Lab L | 가중치 |
|-------|--------|
| L < 12 | 0.0 |
| 12 ≤ L < 20 | 선형 0.5 ~ 1.0 |
| L ≥ 20 | 1.0 |

### 해소된 문제

- `yellow_tree + YELLOW` 탈락 → ✅ 통과 (L 무관 Hue 매칭)
- `yellow_tree + PINK` 오탐 → ✅ 해소 (Chroma < 15 필터)

### 남아있던 한계

K-means가 여전히 BGR 공간에서 수행됩니다. BGR 거리는 인간 시각과 비선형이므로, "사람 눈에 같아 보이는 색"이 아닌 "RGB 값이 가까운 색"끼리 클러스터가 형성될 수 있습니다.

```
dark blue  (50,  0,  0)  ←→  dark red  (0,  0, 50)  :  BGR 거리 = 70  ("비슷하다")
bright yellow (0, 200, 200)  ←→  bright green (0, 200, 0)  :  BGR 거리 = 200 ("다르다")
```

실제로는 어두운 파랑/빨강은 전혀 다른 색이고, 밝은 노랑/녹색은 인접 색입니다. 분석 공간(LCH)과 클러스터링 공간(BGR)이 일치하지 않는 구조적 불일치입니다.

## 4단계: Lab 공간 K-means (현재 구현)

### 변경 내용

이미지 로드 직후 BGR → Lab 변환을 수행하고, 이 Lab 이미지를 K-means에 입력합니다.

```
BGR → Lab 변환 (K-means 전)
    └─ K-means (Lab 공간)
            └─ 클러스터 중심 = Lab 값 (L, a, b)
                    ├─ Hue 계산: atan2(b, a)  (바로 사용, 추가 변환 불필요)
                    ├─ brightnessWeight: Lab L 사용
                    └─ hex 저장: Lab → BGR 역변환
```

이전에는 K-means → 각 중심마다 BGR → Lab 변환을 했습니다. 이제는 한 번만 변환하고 이후 모든 단계가 Lab에서 진행됩니다.

### 효과

`yellow_tree` 기준 비교:

| 항목 | 3단계 (BGR K-means) | 4단계 (Lab K-means) |
|------|---------------------|---------------------|
| YELLOW TargetRatio | ~25% | 52.4% |
| dominant 체크 | 간신히 통과 | 여유롭게 통과 |

Lab 공간에서 클러스터링하니 밝기가 달라도 같은 색으로 보이는 픽셀들이 같은 클러스터로 묶였습니다. HSV V 의존성도 제거되어 파이프라인이 단순해졌습니다.

## 5단계: PURPLE maxHueDist 조정 (30° → 40°)

### 발견한 문제

신규 샘플 `blue_purple.jpg`에서 PURPLE이 탈락했습니다. 이미지의 주색인 짙은 남보라 계열 클러스터들의 Hue가 ~300°로, 당시 PURPLE 범위(335° ± 30° = 305°~365°)에서 5° 벗어났습니다.

```
#101341  31.8%  hue=299.5°  →  PURPLE 거리 35.5° > 30°  탈락 (BLUE에는 매칭)
#22246D  19.3%  hue=300.0°  →  PURPLE 거리 35.0° > 30°  탈락
#5D21BF  11.3%  hue=310.0°  →  PURPLE 거리 25.0° < 30°  매칭 (유일)
```

PURPLE에 매칭된 건 11.3% 클러스터 하나뿐이라 WgtScore=7.55로 임계값(15.0)에 미달했습니다.

### 분석

두 가지 선택지가 있었습니다.

- **refHue 낮추기**: 기준점이 이동해 기존에 높은 점수를 받던 색이 불이익을 받습니다.
- **maxHueDist 늘리기**: 기준점은 그대로, 범위만 확장합니다.

BLUE(285° ± 35° = 250°~320°)와 겹치는 구간이 생기지만, 인접 색의 겹침은 허용하는 설계 방향이므로 문제가 없습니다.

### 변경

`PURPLE(335.0, 30.0)` → `PURPLE(335.0, 40.0)`

새 범위: **295° ~ 375°**. 300° 클러스터들이 포함되어 WgtScore=26.07, TargetRat=38.8%로 통과합니다.
